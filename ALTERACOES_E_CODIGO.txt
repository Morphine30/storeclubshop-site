  O projeto tem duas partes separadas:

  FRONTEND (storyclubshop-site/)
    Arquivos HTML, CSS e JS que o cliente ve no navegador.
    Roda via Live Server na porta 5500.
    Fala com o backend via fetch (chamadas HTTP).

  BACKEND (minha-api/)
    Servidor Node.js com Express.
    Roda na porta 3000.
    Conecta ao MongoDB (banco de dados na nuvem).
    Conecta ao Cloudinary (armazenamento de imagens).
    Conecta ao MercadoPago (pagamentos).

  FLUXO DE UMA COMPRA:
    Navegador → script.js → API Express → MongoDB/Cloudinary/MercadoPago

    CLOUDINARY_API_SECRET → chave privada da API Cloudinary
    MP_ACCESS_TOKEN    → token do MercadoPago (TEST- ou APP_USR-)
    FRONTEND_URL       → URL do site (para liberar CORS)
    PORT               → porta do servidor (3000 local, auto no Render)

┌─────────────────────────────────────────────────────────────┐
│ minha-api/scr/models/Produto.js                             │
└─────────────────────────────────────────────────────────────┘
  O que e:
    Define a "forma" de um produto no banco de dados MongoDB.
    E o equivalente a uma tabela em bancos relacionais (MySQL, PostgreSQL).

  Campos definidos:
    id           → numero unico de identificacao (1, 2, 3...)
    nome         → nome do produto (obrigatorio)
    descricao    → descricao curta (obrigatorio)
    preco        → preco atual em reais (obrigatorio)
    precoAntigo  → preco riscado (opcional)
    parcelas     → texto de parcelamento, ex: "6x de R$16,50"
    estrelas     → avaliacao de 1 a 5
    avaliacoes   → numero de avaliacoes recebidas
    badge        → etiqueta colorida, ex: "🔥 Viral"
    badgeOff     → badge de desconto, ex: "-35%"
    categoria    → Eletrônicos, Beleza, etc
    emoji        → emoji placeholder quando nao tem foto
    imagem       → URL da imagem no Cloudinary (https://...)
    imagemPublicId → ID interno do Cloudinary (para deletar a imagem)
    secao        → onde aparece: "mais-vendidos" ou "novidades"

  Configuracao especial:
    { id: false }      → desativa o id virtual do Mongoose para nao
                         conflitar com o campo id numerico customizado
    toJSON transform   → remove _id e __v das respostas da API
                         (campos internos do MongoDB, nao precisamos ver)

┌─────────────────────────────────────────────────────────────┐
│ minha-api/scr/controllers/produtosController.js             │
└─────────────────────────────────────────────────────────────┘
  O que e:
    O "cerebro" das operacoes de produtos. Cada funcao cuida
    de uma acao diferente (listar, criar, editar, deletar).

  Funcoes:

    listar()
      Busca todos os produtos no MongoDB e retorna em JSON.
      Ordenados pelo campo id (do menor para o maior).

    obter()
      Busca um produto especifico pelo id numerico.
      Retorna erro 404 se nao encontrar.

    criar()
      Pega os dados do formulario (req.body).
      Se tiver imagem, faz upload para o Cloudinary:
        - Recebe o arquivo em memoria (buffer)
        - Converte para base64
        - Envia para o Cloudinary via API
        - Guarda a URL segura (https://res.cloudinary.com/...)
        - Guarda o public_id para poder deletar depois
      Calcula o proximo id (max id atual + 1).
      Salva o produto no MongoDB.

    atualizar()
      Busca o produto pelo id.
      Se chegou nova imagem:
        - Deleta a imagem antiga do Cloudinary (usa imagemPublicId)
        - Faz upload da nova imagem
      Atualiza todos os campos que vieram no formulario.
      Salva no MongoDB.

    deletar()
      Busca o produto pelo id.
      Se tinha imagem, deleta do Cloudinary.
      Remove o produto do MongoDB.

  Funcao auxiliar uploadImagem():
    Recebe o buffer do arquivo e o tipo MIME.
    Converte para base64 no formato data:image/jpeg;base64,...
    Envia para o Cloudinary com a pasta "storeclubshop".
    Retorna o resultado com secure_url e public_id.


┌─────────────────────────────────────────────────────────────┐
│ minha-api/scr/controllers/pagamentoController.js            │
└─────────────────────────────────────────────────────────────┘
  O que e:
    Cria preferencias de pagamento no MercadoPago.

  Como funciona:
    Recebe do frontend: lista de itens do carrinho + URL base do site.
    Monta o objeto de preferencia com:
      items: cada produto com nome, preco e quantidade
      back_urls: para onde redirecionar apos pagar/cancelar/aguardar
      statement_descriptor: nome que aparece na fatura do cartao
      external_reference: ID unico do pedido (timestamp)
    Envia para a API do MercadoPago.
    Retorna ao frontend:
      sandbox_init_point → URL de pagamento em modo teste
      init_point         → URL de pagamento real (producao)

  Seguranca:
    O token do MercadoPago fica no .env, nunca no codigo-fonte.
    Se o token nao estiver configurado, imprime aviso no terminal.


┌─────────────────────────────────────────────────────────────┐
│ minha-api/scr/routes/produtos.js                            │
└─────────────────────────────────────────────────────────────┘
  O que e:
    Define os endpoints (URLs) da API de produtos.
    Configura o multer para aceitar upload de imagens.

  Endpoints criados:
    GET    /api/produtos        → lista todos os produtos
    GET    /api/produtos/:id    → retorna um produto pelo id
    POST   /api/produtos        → cria novo produto (com imagem opcional)
    PUT    /api/produtos/:id    → atualiza produto (com imagem opcional)
    DELETE /api/produtos/:id    → deleta produto

  Configuracao do multer:
    Usa memoryStorage() → guarda o arquivo em RAM (nao salva em disco)
    Limite de 5MB por arquivo
    Aceita apenas arquivos com mimetype "image/*"

  Diferenca da versao anterior:
    Antes: multer.diskStorage() salvava em C:\minha-api\uploads\
    Agora: multer.memoryStorage() guarda em memoria, Cloudinary salva

└─────────────────────────────────────────────────────────────┘
  O que e:
    Define a URL da API de acordo com onde o site esta rodando.

  Como funciona:
    Verifica o hostname do navegador:
      Se for localhost ou 127.0.0.1 → usa http://localhost:3000
      Se for qualquer outro (Netlify) → usa https://SUA-API.onrender.com

  Por que existe:
    Um arquivo HTML nao pode ler variaveis de ambiente.
    Esse arquivo faz o papel de "config dinamica" para o frontend.
    Em desenvolvimento funciona sem mexer em nada.
    Em producao o dev troca so a URL do Render uma vez.

  Onde e incluido:
    Em index.html e admin.html, ANTES do script.js e do script inline.
    Assim window.API_URL ja existe quando os outros scripts rodarem.

 DEPENDENCIAS INSTALADAS (npm)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  cors@2.8.6
    Middleware que adiciona cabecalhos HTTP permitindo que o
    frontend (porta 5500 ou Netlify) acesse o backend (porta 3000).
    Sem isso o navegador bloquearia todas as chamadas.

  multer@2.1.1
    Middleware para receber arquivos (imagens) via formulario HTML.
    Configurado com memoryStorage: guarda em RAM em vez de disco.

  mercadopago@3.1.0
    SDK oficial do MercadoPago para Node.js.
    Usado para criar preferencias de pagamento (Checkout Pro).

  mongoose@9.6.3
    ODM (Object Document Mapper) para MongoDB.
    Permite definir schemas, criar e buscar documentos de forma
    orientada a objetos em vez de usar queries brutas.

  cloudinary@2.10.0

    Usado para fazer upload de imagens e deletar imagens antigas.

  dotenv@17.4.2
    Le o arquivo .env e injeta as variaveis no process.env.
    Permite que o codigo use process.env.MONGODB_URI em vez de
    colocar a senha direto no codigo-fonte.

  nodemon@3.1.14 (desenvolvimento)
    Reinicia o servidor automaticamente quando um arquivo e salvo.
    Usado apenas com npm run dev. Nao vai para producao.
